## 2. Product context

**Iris Code** is an open-source Windows-first Electron desktop GUI that wraps the official Claude Code CLI (`@anthropic-ai/claude-code`). It is currently at v0.4.0. The repository lives at `github.com/oorbitproductivity-oss/iris` and the product site is `iris-code.pages.dev` (Cloudflare Pages).

The app already provides: a Spotlight overlay (global hotkey), multi-agent parallel `claude` subprocesses, sandboxed working directories, per-agent API key picker, an encrypted named-key vault (Electron `safeStorage` with AES-GCM fallback), a command palette, workflow templates, conversation export, a stats dashboard, system notifications, pinned threads + prompt snippets, five themes, global cross-thread search, color-coded tags, voice input, an inline diff viewer, a `CLAUDE.md` memory editor, a per-thread cost tracker, plan mode, an auto-update banner, an interactive welcome tour, and a Telegram remote-agent bridge.

The goal of v0.5 is to close the perceived "premium" gap with **OpenAI Codex** — specifically the in-app browser, integrated terminal, and one-click MCP server installation that Codex users praise. Iris is already feature-rich; v0.5 makes it feel like a workbench rather than a chat window.

The renderer talks to the main process **only** through `preload.js` (`contextBridge.exposeInMainWorld("iris", ...)`). IPC handlers in `main.js` follow the pattern `ipcMain.handle("<namespace>:<verb>", ...)` for request/response and `ipcMain.on("<namespace>:<verb>", ...)` for fire-and-forget. Streaming events are broadcast to all renderer windows + the remote bridge + the Telegram service via the `broadcast(event)` helper in `main.js`.

## 4. Non-negotiable architecture rules

Every feature you implement **must** follow this shape. Reviewers will reject anything that deviates.

1. **Four-layer module pattern.** For each feature `<feature>`, create:
   - `lib/<feature>/index.js` (and supporting files) — main-process logic. Pure Node, no Electron renderer imports.
   - A clearly-labeled block of `ipcMain.handle(...)` / `ipcMain.on(...)` registrations in `main.js`, grouped under a `// ── <Feature> ──` comment.
   - One `<feature>: { ... }` namespace added to `window.iris` in `preload.js`.
   - `app/js/ui/<feature>.js` — renderer UI module. Self-contained, exports an `init(rootEl)` function called from `app/js/app.js`.
2. **Persistent state goes through `Store`** (`lib/store.js`). Do not create new JSON files alongside the store. Add new collections by extending the store API; bump `SCHEMA_VERSION` and write a forward-compatible migration that fills defaults for old data.
3. **Secrets use the existing key vault.** Anything sensitive (API tokens, PATs, OAuth bearers) reuses the `safeStorage` / AES-GCM fallback path used by the named-key vault. Never write secrets to plaintext config files (including `.mcp.json`).
4. **Streaming events use `broadcast(event)`.** The renderer, remote bridge, and Telegram service all subscribe. New `type` values must be additive — old clients should ignore unknown types without crashing.
5. **Feature flags in settings.** Each new feature has a boolean in `store.getSettings()` that defaults to `false` initially (dark-ship). The settings UI gets a toggle. The user enables it after smoke-testing.
6. **No new runtime dependencies without justification.** The only deps in `package.json` today are `express` and `ws`. The **only** approved new runtime deps for v0.5 are:
   - `node-pty` — required for the terminal feature; no Windows-viable alternative.
   - `xterm` + `xterm-addon-fit` — terminal UI in the renderer; pure JS, ~200 KB.
   Anything else requires a one-paragraph justification in the PR description.
7. **No telemetry, no analytics, no new outbound network calls** beyond the single MCP catalog fetch (covered in feature 1). Iris is privacy-respecting by default; document any new outbound call on `site/safety.html`.
8. **Cross-platform first, Windows-tested first.** Iris is Windows-first but ships macOS and Linux builds. Guard platform-specific code with `process.platform` and degrade gracefully — never crash on a missing capability.

### FEATURE 1 — MCP server marketplace (key, v0.5.0)

**Goal.** A settings tab listing curated Model Context Protocol servers. The user clicks "Install globally" or "Install for this agent" and Iris writes the appropriate config so the next agent run picks up the server.

**Files to add**
- `lib/mcp/registry.js` — fetches and caches `https://iris-code.pages.dev/mcp-catalog.json`. Cache for 24 h in `Store`. On fetch failure, return the cached copy; on no-cache failure, return the bundled fallback at `lib/mcp/catalog-bundled.json`.
- `lib/mcp/installer.js` — given a catalog entry + scope (`"global"` | agent id) + collected env vars, write the appropriate `.mcp.json` and validate via `claude mcp list`.
- `lib/mcp/catalog-bundled.json` — fallback catalog shipped with the app.
- `app/js/ui/mcp-marketplace.js` — renderer UI.

**Files to modify**
- `main.js` — add IPC: `mcp:catalog`, `mcp:install`, `mcp:uninstall`, `mcp:list-installed`.
- `preload.js` — expose `iris.mcp = { catalog, install, uninstall, listInstalled }`.
- `lib/agent-manager.js` — when spawning `claude`, pass `--mcp-config <path-to-per-agent-or-global>`.
- `lib/store.js` — add `mcpServers: []` collection, `mcpCatalogCache: { fetchedAt, payload }`.

**Catalog schema** (you author the v1 catalog; ship at minimum: Playwright, GitHub, Postgres, Fetch, Filesystem, Memory, Sequential-thinking, Time):

```json
{
  "version": 1,
  "updated": "2026-05-24T00:00:00Z",
  "servers": [
    {
      "id": "playwright",
      "name": "Playwright Browser",
      "description": "Drive a real Chrome browser from the agent.",
      "category": "browser",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-playwright"],
      "env": []
    },
    {
      "id": "github",
      "name": "GitHub",
      "description": "Read issues, PRs, repos, code search.",
      "category": "vcs",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": [{ "name": "GITHUB_TOKEN", "required": true, "secret": true, "label": "Personal access token" }]
    }
  ]
}
```

**Secrets handling.** Required `secret: true` env vars are entered in the install dialog, stored in the encrypted key vault under id `mcp:<serverId>:<envName>`, and injected at agent-spawn time via the `env` field of the `claude` child-process — never written to `.mcp.json` on disk.

**Acceptance criteria**
- Open Settings → MCP tab → see ≥ 8 servers.
- Install GitHub globally with a PAT → create a new agent → ask "list my open PRs" → agent uses the MCP tool.
- Disable feature flag → MCP tab hidden, installed configs left intact.
- Catalog fetch fails (offline) → UI shows cached or bundled catalog, no crash.

**Test**: `tests/mcp-marketplace.test.js` — mock the catalog fetch, install + uninstall a synthetic server, assert `.mcp.json` correctness.

**Docs**: `docs/mcp-marketplace.md` — user-facing how-to.

### FEATURE 2 — Embedded browser pane (key, v0.5.0)

**Goal.** A right-side `<webview>` that the user can open per agent. The agent can navigate, click, and screenshot it via the Playwright MCP server (installed via feature 1). The user can also drive it directly with the mouse — the same browser instance the agent sees.

**Files to add**
- `lib/browser/playwright-bridge.js` — manages the lifecycle of `npx @modelcontextprotocol/server-playwright` per agent. Lazy-spawned when the pane opens; killed when it closes for > 5 min.
- `app/js/ui/browser-pane.js` — split-pane component with draggable splitter; remember ratio per-agent in settings.

**Files to modify**
- `main.js` — IPC: `browser:open`, `browser:close`, `browser:navigate`, `browser:screenshot-to-thread`.
- `preload.js` — `iris.browser = { open, close, navigate, onScreenshot }`.
- `app/index.html` — add the split-pane container under the chat view.
- `app/css/*.css` — split-pane styles consistent with the active theme.

**Implementation notes**
- Use Electron's `<webview>` tag with `partition="persist:agent-<id>"` for per-agent cookie isolation.
- `nodeIntegration: false`, `contextIsolation: true`, `allowpopups: false`. Apply a strict CSP on the host page (`default-src 'self'`).
- Cap at one webview per agent. Hide (not destroy) when the pane is collapsed so cookies persist. Destroy after 5 min hidden to reclaim memory.
- Screenshots captured by the agent stream into the chat as inline `<img src="data:image/png;base64,...">` reusing the diff-viewer's image rendering CSS classes.

**Acceptance criteria**
- Toggle pane on a new agent → see a blank webview.
- Ask "go to example.com and tell me the heading" → webview navigates visibly, agent screenshots, replies with the heading.
- Close + reopen pane → cookies persist; navigation history preserved.
- Open browser pane on 5 different agents → no cross-agent cookie bleed.

**Test**: `tests/browser-pane.test.js` — smoke test that `playwright-bridge` spawns + kills cleanly.

**Docs**: `docs/browser-pane.md`.

### FEATURE 3 — Integrated terminal (key, v0.5.0)

**Goal.** A terminal tab per agent (or several) backed by a real PTY, so the user can `git status`, run dev servers, etc. without alt-tabbing. Optionally share the last N lines of output with the agent.

**Files to add**
- `lib/terminal/pty-manager.js` — wraps `node-pty`. Tracks `{ id, agentId, cols, rows, cwd, shell, history }`. Caps history at 10 000 lines per terminal.
- `app/js/ui/terminal-pane.js` — xterm.js host. Tab strip for multiple terminals per agent.

**Files to modify**
- `main.js` — IPC: `term:spawn`, `term:write`, `term:resize`, `term:kill`, plus broadcast `term:data` events.
- `preload.js` — `iris.terminal = { spawn, write, resize, kill, onData }`.
- `package.json` — add `node-pty`, `xterm`, `xterm-addon-fit` to `dependencies`; add `electron-rebuild` to `devDependencies` and an `"electron-rebuild"` line in the build script chain.
- `DEPLOY.md` — document the `electron-rebuild` step for releases.

**Implementation notes**
- Default shell: Windows → prefer `pwsh.exe`, fall back to `powershell.exe`, then `cmd.exe`. macOS/Linux → `process.env.SHELL || /bin/bash`.
- One PTY per `(agentId, terminalId)` pair. PTYs survive renderer reload.
- "Share with agent" button copies the last N (default 200) lines from history into the chat input as a fenced code block — explicit, never automatic, to keep token spend predictable.
- If `node-pty` fails to load (prebuilt binary missing), disable the feature with a clear toast — do **not** crash the app or block startup.

**Acceptance criteria**
- Open terminal tab → run `git status` → see output streaming.
- Resize the pane → terminal reflows.
- Close + reopen the renderer → terminals reattach (PTYs survived).
- Click "share last 50 lines" → those lines appear in the chat input ready to send.

**Test**: `tests/terminal.test.js` — spawn a PTY, write `echo hi`, assert output.

**Docs**: `docs/terminal.md`.

### FEATURE 4 — Custom slash commands (nice-to-have, v0.5.1)

**Goal.** Let users define `/standup`, `/fix-pr`, `/journal`, etc. — typing the trigger in any chat expands a template prompt with `{{cursor}}` and `{{selection}}` placeholders.

**Files to modify**
- `lib/store.js` — add `slashCommands: []` collection. Each: `{ id, trigger, name, description, prompt, scope: "global" | <agentId> }`. Triggers are case-insensitive, must match `/[a-z0-9_-]+/`.
- `app/js/ui/slash-commands.js` — already exists; extend to:
  - Show the user's custom commands alongside built-ins.
  - Provide CRUD UI: list, add, edit, delete, import/export JSON.
- `app/js/ui/snippets.js` — share the trigger-detection logic; do not duplicate.

**Acceptance criteria**
- Define `/standup` → type it in any thread → expands.
- `{{selection}}` is replaced with the currently-selected text in the input; `{{cursor}}` becomes the final cursor position.
- Export a command as JSON → import into a fresh install → it round-trips.

**Test**: extend `tests/slash-commands.test.js`.

### FEATURE 5 — Per-thread cost budgets (nice-to-have, v0.5.1)

**Goal.** Set a $ ceiling per thread. Warn at 80%, hard-pause at 100% with "raise / stop" modal.

**Files to modify**
- `lib/store.js` — per-thread fields: `costBudgetUsd: number | null`, `costBudgetAction: "warn" | "pause"` (default `"pause"`).
- `lib/agent-manager.js` — after each turn's cost calc (already exists), emit `cost:warn` at ≥ 80% and `cost:exceeded` at ≥ 100%. On `cost:exceeded` with `action === "pause"`, refuse to send the next user message until the budget is raised or cleared.
- `app/js/ui/cost-tracker.js` — already exists; extend to:
  - Show a thin progress bar inside the cost pill (orange ≥ 80%, red ≥ 100%).
  - Toast on `cost:warn`. Modal on `cost:exceeded` with "Raise to $X" / "Stop agent" / "Continue once" buttons.

**Acceptance criteria**
- Set $0.50 budget → run an expensive task → warn at $0.40, pause at $0.50.
- Raise to $1 → agent continues seamlessly.
- Clear budget → behaves as today.

**Test**: extend `tests/cost-tracker.test.js`.

### FEATURE 6 — Web search results as rich cards (nice-to-have, v0.5.2)

**Goal.** When the Iris orchestrator or an agent emits a list of web sources, render them as cards (favicon, title, snippet, "open" / "open in pane" buttons) instead of plain markdown links.

**Files to modify**
- `lib/router/prompts.md` — append one paragraph teaching the orchestrator: "When citing web sources, wrap each in `<iris-web-result url=... title=...>snippet</iris-web-result>`; the UI will render them as cards. Plain links still work for inline references."
- `app/js/lib/markdown.js` — detect the custom tag during markdown render; replace with a card DOM node. Use `https://www.google.com/s2/favicons?sz=32&domain=<host>` for favicons (already an outbound call the user expects from a web result; document on `site/safety.html`).
- `app/css/*.css` — card styles (one variant per theme).

**Behavior when the browser pane (feature 2) is enabled**: cards show a second button "Open in pane" that calls `iris.browser.open(agentId, url)`.

**Acceptance criteria**
- Ask Iris a question that triggers web search → answer renders as 3–5 source cards.
- Disable the feature flag → degrades to plain markdown links.
- Click "open" → opens in default browser via `shell:openExternal`.

**Test**: extend `tests/markdown.test.js` with a sample `<iris-web-result>` block.

### FEATURE 7 — Native Mica/Acrylic chrome (nice-to-have, v0.5.2)

**Goal.** Opt-in translucent window backdrop on Windows 11 and macOS. Pure visual polish.

**Files to modify**
- `main.js` `createMainWindow()` — when `settings.translucentChromeEnabled` is true and the platform supports it:
  - Win11 (`process.platform === "win32"` and `parseInt(os.release().split(".")[2]) >= 22000`): add `backgroundMaterial: "mica"`.
  - macOS: add `vibrancy: "under-window"` and `visualEffectState: "active"`.
  - Other platforms / older Windows: ignore the flag; current opaque behavior.
- `app/js/ui/theme-picker.js` — add a toggle: "Translucent window (Windows 11 / macOS)". Disabled with a tooltip on unsupported OSes.
- `app/css/themes/codex-dark.css` and `midnight.css` — when `body[data-translucent="true"]`, swap solid panel backgrounds for `rgba(...)` with appropriate alpha (test contrast against busy wallpapers — aim for WCAG AA on body text).

**Acceptance criteria**
- Win11 user enables toggle → window picks up wallpaper softly behind tinted panel.
- macOS user enables toggle → vibrancy active.
- Win10 user sees the toggle disabled with a tooltip.
- Toggle off → identical to current appearance.

**Test**: visual; document a manual checklist in `docs/themes.md`.

## 8. Commit and PR conventions

- **Commit author**: `oorbitproductivity-oss <oorbitproductivity@gmail.com>`. Never use the user's personal name or email.
- **Commit messages**: imperative, present tense, scoped: `mcp: add catalog fetcher with 24h cache`, `browser: per-agent webview partition`, etc.

- **Run `npm test` before every push.** All existing tests must still pass; new tests must accompany new code.

## 9. Testing approach

- Smoke tests live in `tests/` and are wired into `npm test`. The existing `tests/run-all.js` is the entry point.
- Each new feature adds one `tests/<feature>.test.js`. Use the same lightweight assertion style the existing tests use (no new test framework).
- For UI-only changes (Mica chrome, card rendering), include a manual test checklist in the matching `docs/<feature>.md`.
- Before opening the v0.5.0 PR, run `npm run dist:dir` and launch the unpacked build at least once to confirm the native `node-pty` module is rebuilt correctly.

## 10. Things you must not do

- Do not rewrite the existing agent manager, store, or router. Extend them.
- Do not introduce TypeScript, a bundler, React, or any UI framework. The app is plain ES modules + plain DOM today. Keep it that way.
- Do not add telemetry, crash reporting, or analytics of any kind.
- Do not commit secrets, even in tests. Use placeholder values like `"sk-test-..."`.
- Do not mark a feature "done" until its acceptance criteria are checked on a fresh build.
- Do not modify `site/safety.html` to claim audits or counts that did not happen. If you add a new outbound call (the MCP catalog fetch, the favicon fetch), describe it factually with a link to the relevant code.
